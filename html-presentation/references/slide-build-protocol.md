# Slide Build Protocol

## Phase 2 — Parallel Batch Build

Each slide's HTML is generated by its own subagent — giving every slide a full context window. The main context orchestrates: launch batches, collect returned HTML, insert into the file, verify, repeat.

**Architecture: Generate-then-Insert**

Slide subagents do NOT edit the HTML file directly. They **generate and return** the slide HTML as a string. The main agent then inserts each returned slide into the file sequentially using the `edit` tool. This eliminates concurrent file write conflicts while keeping the main context free of large HTML content.

**BATCH_SIZE = 4** — adjust to 3 for decks with heavy SVG, 5 for simple decks.

---

### Phase 2 Loop

**Before starting**: confirm `generate_shell.py` succeeded (Step 3.5 passed) and all N `<!-- INSERT_SLIDE_N -->` markers are present in the HTML file.

**For each batch of BATCH_SIZE slides:**

#### Step A — Launch batch subagents in parallel

Launch all BATCH_SIZE subagents **simultaneously** in a single message (multiple `runSubagent` calls in one response). Each subagent targets ONE slide only.

**Subagent type:** `general-purpose` | **readonly:** `false`

Each subagent prompt MUST include:

- **Task** (exactly): "Generate the HTML for slide N only. Write the complete `<div id='sN' class='slide'>...</div>` block to `{deck_folder}/drafts/slide_N.html` using the `write` tool. Do NOT return the HTML as text. Do NOT edit the main deck file. Do NOT include any `<!-- INSERT_SLIDE_ -->` markers."
- **Canonical classes rule** (mandatory): "Use canonical CSS component classes for their container type — these classes are already defined in the shell CSS:
  - `.card-grid` — any grid of card elements (2–4 columns)
  - `.card` — individual card elements inside a grid
  - `.icon-list` — icon+label `<ul>` elements
  - `.code-block` — syntax-highlighted code blocks
  - `.step-flow` — step-with-arrows layouts
  Use these classes instead of duplicating their properties as inline `style=` attributes. Inline styles are only for values that differ from the class defaults (e.g. a custom `grid-template-columns` ratio, a specific `border-left` color, overriding centered text alignment)."
- Slide plan entry for THIS SLIDE ONLY (e.g., `Slide 5: Core — Architecture Diagram showing the data flow`)
- Full slide plan for narrative context
- All 9 reference file contents inline (embed full text): `presentation-runtime.md` (most critical — copy exactly), `slide-structure.md`, `content-rules.md`, `html-output-spec.md`, `visual-components.md`, `slide-build-protocol.md`, `css-animations.md`, `accent-colors.md`, `graphics-embedding.md`. If combined content too large, abbreviate `css-animations.md` and `accent-colors.md` to first 50 lines.
- Relevant SKILL.md sections inline: Shell Safety Rules, Presenter/Agenda/BLUF rules — required for structural slides
- HTML file path (for reference only — subagent does NOT edit it)
- Working directory path (skill root where `scripts/` lives) — for running `svg_calc.py`
- Config values: `accent_color`, `speaker_notes`, `exec_mode`, `customer`, `industry`
- **Shell Safety Rules** (mandatory — subagents do NOT inherit SKILL.md):
  - NEVER use heredoc syntax (`<<EOF`) or multi-line `python3 -c "..."` in bash
  - Use `write` tool to create temp file, then run it: `python3 /tmp/script.py`
- **Output format** (required, exact):
  ```
  Write to {deck_folder}/drafts/slide_N.html:
  <div id="sN" class="slide">
    ...slide content...
  </div>
  ```

#### Step B — Insert batch drafts

After ALL batch subagents complete (each has written its draft to `{deck_folder}/drafts/slide_N.html`):

For each slide N in the batch (ascending order), run:
```bash
python scripts/run_script.py insert_slide.py <deck.html> {deck_folder}/drafts/slide_N.html N
```

`insert_slide.py` extracts the `<div id="sN" class="slide">` block from the draft, replaces the `<!-- INSERT_SLIDE_N -->` marker in the deck, and writes the deck back.

Insert slides sequentially — do NOT run multiple `insert_slide.py` calls in parallel.

On `ERROR:` output: retry the subagent once (same prompt), re-run `insert_slide.py`. After 2 failures, build the slide in main context as fallback.

#### Step C — Verify batch

After inserting all slides in the batch:

1. Grep HTML for any remaining markers from this batch: none of `<!-- INSERT_SLIDE_N -->` (for N in batch) should remain
2. `insert_slide.py` exits non-zero on any error — check output for `SUCCESS:` on each call
3. If `ERROR:` printed → retry that slide individually (re-run subagent, re-run insert_slide.py)
4. After 2 failed individual retries, build the slide in main context as fallback

#### Step D — Update progress

After each successful batch:

1. Update `manifest.json`: `html: {status: "building", last_slide: N}` where N = highest slide in batch
2. Report: `Slides 1–4 complete (4/[total]). Building slides 5–8...`

After final batch: `html: {status: "complete", last_slide: [total]}` and `All [total] slides complete.`

### Resuming After Interruption

If HTML file already exists when Phase 2 starts, find all remaining `<!-- INSERT_SLIDE_N -->` markers. Build batch list from those markers only (skip completed slides). Inform user: "Resuming from slide N."

## Post-Build Processing

> [CONDITION: skip the embed phase (step 1 below) if HTML contains no `{{...}}` placeholder tokens; skip the QR phase (step 2 below) if deck has no `<a>` links to external URLs]

**Post-build processing (when deck includes user-provided images, SVGs, or Snowflake logo):**
1. **Embed phase** — Write HTML with placeholder tokens during Phase 1 and 2. Do NOT write `<svg>`/`<path>` markup or base64 during those phases.
   - `{{SNOWFLAKE_LOGO}}` — standalone token for Snowflake logo on title slide
   - `{{SVG_INLINE:path}}` or `{{SVG_INLINE:path|css-style}}` — standalone token for diagram/chart SVG files
   - `{{LOGO_INLINE:path}}` or `{{LOGO_INLINE:path|css-style}}` — standalone token for logo/decorative SVG (skips geometry validation)
   - `{{IMG:path}}` or `{{IMG:path|max-px}}` — inside `<img src="...">` for raster images
   After all slides built, inform user: "Embedding images — this may take a moment." Run `python scripts/run_script.py embed_image.py <deck.html>`. Happens entirely in Python — no SVG or base64 enters conversation context.
2. **QR appendix phase** — If deck has `<a>` links to external URLs, inform user: "Generating QR code appendix." Run `python scripts/run_script.py generate_qr_appendix.py <deck.html>`. Auto-numbers new slide and updates counter. **Requires `<div class="counter"></div>` in HTML** — MUST be in initial build (see `references/html-output-spec.md` > Navigation).

If no user-provided images, no SVGs, no Snowflake logo (only Material Symbols, inline CSS), skip embed phase — HTML is already self-contained.

**3. URL validation phase** — After QR appendix generated, verify all external links resolve. Run:
```bash
python scripts/run_script.py validate_urls.py [folder]/[topic-slug]-[audience-slug]-slides.html
```
Any non-200 URL MUST be replaced before proceeding. After fixing, re-run `generate_qr_appendix.py` — idempotent, rebuilds QR codes. Do not proceed to Step 4 with broken links.

**Bot-detection fallback:** If `validate_urls.py` reports `403-bot-blocked`, verify via `web_fetch`. If content returned, URL valid. Only replace URLs failing both methods.

**4. Icon subsetting phase** — After all other post-build steps are complete, optimise the Material Symbols font payload. Run:
```bash
python scripts/run_script.py subset_icons.py [folder]/[topic-slug]-[audience-slug]-slides.html
```
This rewrites the Google Fonts `<link>` to request only the icons actually used in the deck (`&icon_names=...`), reducing the font from ~295 KB to ~2 KB. Run last so all icon spans are present.

For image/graphics technical details, see `references/graphics-embedding.md`.

## Replacing a Fully-Built Slide

Use `replace_slide.py` when a slide has already been inserted into the deck (no `<!-- INSERT_SLIDE_N -->` marker remains) and needs to be rebuilt in-place.

```bash
python scripts/run_script.py replace_slide.py <deck.html> <draft.html> <slide_number>
```

- **`deck.html`** — the destination deck; modified in-place
- **`draft.html`** — file containing the replacement slide (single-slide draft or full deck)
- **`slide_number`** — integer matching the `id="sN"` attribute (e.g. `7` for `id="s7"`)

The script uses depth-tracking HTML parsing to locate the exact `<div id="sN" class="slide">...</div>` block (handles any number of nested `<div>` elements) and replaces it with the matching block from the draft.  It handles both attribute orderings (`id` first or `class` first).

**Workflow:**

1. Have a subagent write the replacement slide to `{deck_folder}/drafts/slide_N.html`
2. Run `replace_slide.py` to swap it into the deck
3. Re-run `validate_deck.py` to confirm no regressions

**`replace_slide.py` vs `insert_slide.py`:**

| Scenario | Use |
|---|---|
| Slide not yet inserted (`<!-- INSERT_SLIDE_N -->` marker present) | `insert_slide.py` |
| Slide already inserted — need to rebuild it | `replace_slide.py` |

## Slide ID and Numbering Rules

- Every slide (including Agenda and Presenter) MUST have sequential `id="sN"` starting from `s1` for title slide.
- Agenda and Presenter slides are real slides with their own IDs, even though they don't count toward content slide plan.
- `<span id="total">` MUST equal actual slide div count. JS init MUST include `document.getElementById('total').textContent = slides.length;` before `show(0)` — self-corrects at runtime, enforced by `validate_deck.py` Check #2.
- When adding/removing slides, renumber ALL subsequent IDs and update total. **Always renumber in reverse order** (highest first) to avoid double-replacement bugs. Use `scripts/insert_presenter.py` via `run_script.py` for post-build presenter injection (handles resize, base64, insertion, renumbering, total update).
